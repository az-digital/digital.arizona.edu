Our Process

The Arizona Digital development process has been continuously refined to ensure that every asset that we share with the university is secure, accessible, performant and maintainable.

The entire process has been thoroughly reviewed by the Auditor General of the State of Arizona.

How it works

1. New Issue Created

The process begins with an issue in GitHub to document the intended change. The issue is linked to the pull request, which is updated throughout the process until the change is completed.

When we are notified of a Drupal.org Security Advisory, critical security updates are prioritized according to the Drupal.org security risk level. The process is designed for rapid deployment of highly critical updates to ensure they follow the same basic process as other code updates, though processed in an accelerated time frame. Timing for processing Drupal updates is outlined in the Campus Web Services Security Maintenance Timeframe Matrix on Confluence.

2. Feature Branch Created

A local copy of the Quickstart codebase is downloaded by the Quickstart Developer, and a new issue branch is created. Development never happens directly on the Main branch.

3. Local Development Completed

The Developer works on the issue and tests their changes in their local development environment, on their own computer. This allows the developer to ensure that their code is fully functional in their environment before submitting it for review. Developers adhere to coding best-practices, and Drupal.org Coding Standards.

4. Pull Request Created/Updated

Once the Developer considers the issue to be ready for review they submit a pull request in GitHub which kicks off the review process. Only Developers in our GitHub organization are able to submit pull requests. Developers can only be added to the organization by Arizona Digital Admins.

5. Complete Testing

The pull request goes through several types of testing and review.

5a. Automated Testing Completed

Automated testing is completed using GitHub Actions and ProboCI. Every pull request automatically initiates our Continuous Integration (CI) process. This runs our automated tests, completes the entire build process and makes a functional website with the proposed changes available for review. Tests are all documented in YAML files within the codebase and include:

  • Static dependency scanning Static dependency scanning is run as a part of the Continuous Integration (CI) pipeline (via sensiolabs/security-checker for composer dependency scanning).
  • Build and install testing Quickstart is built and installed in a temporary review environment, which tests the full build and install process (using ProboCI).
  • Behavioral testing Behat behavioral tests are run to check for regressions.

5b. Code Review Completed

Developers review the code and provide recommendations. Bitbucket highlights all of the proposed changes to the source code directly on the pull request and provides an interface for comments and discussion. All activity on the pull request is also broadcast to a community Slack channel for visibility. The primary focus of this review is:

  • Coding standards review following Drupal.org coding standards.
  • Security best practices review by at least one Quickstart Admin, with attention to OWASP Top 10, which identifies the most critical security risks to web applications. The review will also attempt to identify applicable risks outlined in the ISO-1300-S1 Application Security Standard and the ISO-1300 Secure Application Development and Administration Policy.
  • Quickstart Admin training All Quickstart Admins are required to complete the annual Secure Web Application Development Training, and follow the Acceptable Use for System Administrators Policy.

5c. QA Review Completed

Once the test website is built on ProboCI, a link is automatically made available, so that both developers and non-developers are able to review the proposed changes in an actual website. Manual website review & testing is then completed by Quickstart Admins and other Developers using the ProboCI Build. The primary focus of this review is to test the changes in a website that provides a visual representation of how the changes will appear once applied to an actual website. The website is reviewed and tested with the following focus.

  • Quality Assurance (QA) & User Experience (UX) review
  • Functional testing
  • Accessibility review
  • Regression testing

Changes Approved

The pull request is approved by at least two developers, one of whom is an Arizona Digital Admin. (Each Arizona Digital Admin has completed the ISO security training and reviewed the Secure Application Development Standard.)

If any of the phases of review uncover issues that need to be fixed, the developer will go back to their local development environment to make the changes and update their pull request to go back through the review process before it can be merged.

6. Changes Merged

Once the changes are approved, they are then merged into the Main branch by an Arizona Digital Admin. (Only Admins are authorized to merge.)

7. Dev Site Updated

The associated development website (demo.quickstart.arizona.edu) is then updated to provide a fully built functioning website with the latest codebase. Dynamic web application scanning is run weekly using Tenable Web Application Scanner (WAS).

8. New Version Released

A new artifact bundle is built and made available on GitHub for download by campus units. A new release of Quickstart (or Arizona Bootstrap), including the changes, is tagged. Each tagged release remains available for download by version number. Developers using the Quickstart codebase are notified of the new release via the Quickstart Slack channels associated with Arizona Digital. If the update is associated with a critical security release, timely notifications are also sent through the universityâs UAWeb, Drupal, and ITSM Listservs.

9. Quickstart Upstream Updated

An artifact repo is created with a fully built codebase, which is made available as the Quickstart Upstream on the Pantheon environment managed by Campus Web Services. Campus Web Services uses this Pantheon Upstream to create and update their managed Arizona Sites. Campus Web Services also makes this upstream available for campus partners who are in the Pantheon environment managed by Campus Web Services. Pantheon provides an added layer of security, as documented at pantheon.io/security.

Change Ticket Closed

The associated issue is closed automatically when the pull request is merged.

Get Involved

A team of web-focused volunteers known as Arizona Digital meets weekly to build and test products like Arizona Bootstrap and UA Quickstart.

  • Join the Arizona Digital discussions on Slack
  • Submit pull requests on GitHub
